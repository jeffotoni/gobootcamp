go test

Pacotes de teste: 


$ go test [build/test flags] [packages] [build/test flags & test binary flags]

O teste Go automatiza o teste dos pacotes nomeados pelos caminhos de importação. Ele imprime um resumo dos resultados do teste no formato: 


ok   archive/tar   0.011s
FAIL archive/zip   0.022s
ok   compress/gzip 0.033s

Seguido por uma saída detalhada para cada pacote com falha. 'Go test' recompila cada pacote junto com quaisquer arquivos com nomes que correspondam ao padrão de arquivo "* test.go". Esses arquivos adicionais podem conter funções de teste, funções de benchmark e funções de exemplo. Veja 'go help testfunc' para mais informações. Cada pacote listado causa a execução de um binário de teste separado. Arquivos cujos nomes começam com "" (incluindo "_test.go") ou "." são ignorados. Os arquivos de teste que declaram um pacote com o sufixo "_test" serão compilados como um pacote separado e, em seguida, vinculados e executados com o binário de teste principal. A ferramenta go irá ignorar um diretório chamado "testdata", tornando-o disponível para conter os dados auxiliares necessários para os testes. Como parte da construção de um binário de teste, go test executa go vet no pacote e seus arquivos de origem de teste para identificar problemas significativos. Se go vet encontrar algum problema, vá testá-los e não executará o binário de teste. Apenas um subconjunto de alta confiança das verificações de veterinário padrão é usado. Esse subconjunto é: 'atomic', 'bool', 'buildtags', 'nilfunc' e 'printf'. Você pode ver a documentação desses e de outros testes veterinários em "go doc cmd / vet". Para desativar a execução do go vet, use a sinalização -vet = off. Todas as saídas de teste e linhas de resumo são impressas na saída padrão do comando go, mesmo se o teste as imprimiu com seu próprio erro padrão. (O erro padrão do comando go é reservado para erros de impressão na construção dos testes.) O teste Go é executado em dois modos diferentes: O primeiro, denominado modo de diretório local, ocorre quando go test é invocado sem argumentos de pacote (por exemplo, 'go test' ou 'go test -v'). Nesse modo, go test compila as fontes e testes do pacote encontrados no diretório atual e, em seguida, executa o binário de teste resultante. Neste modo, o armazenamento em cache (discutido abaixo) é desabilitado. Após a conclusão do teste do pacote, go test imprime uma linha de resumo mostrando o status do teste ('ok' ou 'FAIL'), o nome do pacote e o tempo decorrido. O segundo, chamado modo de lista de pacotes, ocorre quando go test é invocado com argumentos de pacote explícitos (por exemplo 'go test math', 'go test. /...' e até mesmo 'go test.'). Neste modo, faça um teste de compilação e teste cada um dos pacotes listados na linha de comando. Se um teste de pacote for aprovado, go test imprime apenas a linha de resumo final 'ok'. Se um teste de pacote falhar, go test imprime a saída de teste completa. Se chamado com o sinalizador -bench ou -v, go test imprime a saída completa mesmo para passar nos testes de pacote, a fim de exibir os resultados de benchmark solicitados ou registro detalhado. Além dos sinalizadores de construção, os sinalizadores controlados pelo próprio 'go test' são: 


-args
    Pass the remainder of the command line (everything after -args)
    to the test binary, uninterpreted and unchanged.
    Because this flag consumes the remainder of the command line,
    the package list (if present) must appear before this flag.

-c
    Compile the test binary to pkg.test but do not run it
    (where pkg is the last element of the package's import path).
    The file name can be changed with the -o flag.

-exec xprog
    Run the test binary using xprog. The behavior is the same as
    in 'go run'. See 'go help run' for details.

-i
    Install packages that are dependencies of the test.
    Do not run the test.

-json
    Convert test output to JSON suitable for automated processing.
    See 'go doc test2json' for the encoding details.

-o file
    Compile the test binary to the named file.
    The test still runs (unless -c or -i is specified).

Para mais informações sobre sinalizadores de construção, consulte 'go help build'. Para obter mais informações sobre como especificar pacotes, consulte 'go help packages'. O pacote de teste é executado lado a lado com o comando go test. O teste do pacote deve ter o sufixo "_test.go". Podemos dividir os testes em vários arquivos seguindo esta convenção. Por exemplo: "myprog1_test.go" e "myprog2_test.go". Devemos colocar nossas funções de teste nesses arquivos de teste. Cada função de teste é uma função pública exportada cujo nome começa com "Teste", aceita um ponteiro para um objeto testing.T e não retorna nada. Como isso: Exemplo one/myprog1_test:

  go install Homepage
